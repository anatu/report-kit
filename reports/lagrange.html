Lagrange: agentic loops as constrained optimization

Lagrange treats an agent loop as constrained optimization — maximize task success subject to explicit budgets and hard limits, each constraint priced by a Lagrange multiplier the loop tunes as it runs. The kernel is stdlib-only; a Claude Agent SDK backend and a checkpoint layer sit on top. This page tracks the design and where it's going.

Subsystems

Every agent loop is the same skeleton

From AutoGPT through ReAct, Reflexion, Self-Refine, and GEPA, the structure repeats: generate → evaluate → select → repeat. The only real variables are where the scoring signal comes from and what decides "stop." The universal failure mode is termination — letting the model decide it's "done" in natural language, which drifts, loops, or over-refines.

Lagrange makes those two things first-class. A loop is a small, explicit state machine with pluggable slots; the kernel owns the parts every framework gets wrong — convergence, constraints, budget, and observability.

The name is literal: constraints carry a price

A coding loop that maximizes success subject to budgets (tokens, cost, wall-clock) and hard limits (tests must pass) is a constrained optimization problem. Lagrange optimizes the Lagrangian directly:

L = success − Σi λi · max(0, costi − limiti)

Each constraint owns a multiplier λ — the price of violating it — updated from measured violation by a small PID controller (the integral term alone is classic dual ascent). The penalized objective feeds selection and plateau detection, so over-budget candidates lose rank without being hard-banned.

Honest about the metaphor

The constraint-pricing is literal and load-bearing. The full controller-theory machinery (stability guarantees, auto-converging λ) is not — λ is sensitive and task-dependent. Lagrange ships the controller and Pareto-sweep tooling, not a promise that one knob auto-tunes itself.

The controller engages, then relaxes

On a toy hill-climb (drive x to 7.3, with each step's jump constrained to ≤ 2.0), the loop converged in 35 iterations. The figure shows the mechanism working: when the agent proposes a big jump, λ rises and the penalized objective drops below the raw value; as steps shrink, λ relaxes toward zero. The stop algebra refuses underspecified loops

Stopping is a composable algebra. Conditions combine with |; the first to fire wins and records why, and whether that counts as convergence.

stop = Success() | Plateau(patience=3) | Oscillation() | MaxIters(20)

Every condition advertises two facts: whether it can converge (success) and whether it guarantees termination (a hard cap). The loop reads them and refuses to run a config that can't do both — a loop that can only stop by exhausting its budget is the lineage's most expensive anti-pattern.

Table 1. Stop conditions by role.
RoleConditionsConverges?
SuccessSuccess, ValueThreshold, Customyes
StagnationPlateau, Oscillationno*
Hard capMaxIters, WallClock, TokenBudget, CostBudgetno
ControlPause (HITL), ConstraintViolatedno

* Plateau can be marked convergent when a plateau means "good enough."

One query() call is one loop step

The default backend drives a Claude agent. One Lagrange step is one query() turn; the executor chains session_id across iterations so context carries, and feeds a failing verdict back as the next prompt — a Reflexion-style refine cycle on a real agent. Per-turn caps map to the SDK's max_turns / max_budget_usd; each turn's usage is recorded into the loop so TokenBudget / CostBudget bound the whole run.

Show the coding-loop in full 
from lagrange import Budget
from lagrange.backends.claude import ClaudeExecutor
from lagrange.stop import Success, Plateau
from lagrange.verify import CommandVerifier

executor = ClaudeExecutor(
    task="The tests are failing. Fix the code and run `pytest -q`.",
    goal="pytest passes",
    cwd="/path/to/project",
    allowed_tools=["Read", "Edit", "Bash"],
    per_step_max_turns=30,          # SDK hard stop per turn
)

loop = executor.build_loop(
    verifier=CommandVerifier(lambda _: ["pytest", "-q"], cwd="/path/to/project"),
    stop=Success() | Plateau(patience=3),
    budget=Budget(max_iters=8, max_cost_usd=10.0),
)
result = loop.run()             # result.best.payload.text == final agent output

Two verifier styles cover the spectrum. CommandVerifier runs a real command and gates on the exit code — deterministic and transcript-provable. ClaudeJudge is the actor/critic split: a fresh, cheap model judges the agent's surfaced output. The judge can't run tools, so the goal must be phrased to be provable from what the agent showed.

One checkpoint primitive: resume, time-travel, HITL

Give a loop a run_id and a Persister and it checkpoints every iteration. The same primitive serves three needs:

  • Crash recovery — restart, call run() again; the loop restores iteration, archive, memory, λ state, usage, and the agent's session_id, then continues.
  • Time-travel — every iteration is kept; history(run_id) lists them, load(run_id, n) loads any.
  • Human-in-the-loop — a Pause condition checkpoints and exits; release it and resume.

Two stores ship: MemoryPersister (in-process) and SQLitePersister (durable, stdlib-only, survives a crash). The Claude executor auto-registers as a state provider, so the agent session is checkpointed without extra wiring.

Limitations

  • SQLitePersister uses pickle — never load checkpoints from an untrusted source.
  • The ClaudeJudge verifier can't run tools; convergence conditions it judges must be provable from the agent's surfaced output.
  • Each step bridges sync↔async with asyncio.run; it raises if called from inside a running event loop (async API is future work).
  • λ is sensitive and task-dependent — the controller helps, but tuning is still empirical.
