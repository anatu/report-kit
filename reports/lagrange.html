Lagrange: agentic loops as constrained optimization

An agent loop as constrained optimization. Maximize task success under explicit budgets. Each constraint is priced by a Lagrange multiplier, tuned as the loop runs. Stdlib-only kernel; Claude backend and checkpointing on top.

Subsystems

Every agent loop is the same skeleton

AutoGPT, ReAct, Reflexion, Self-Refine, GEPA — one shape: generate → evaluate → select → repeat. Two variables: where the score comes from, and what stops the loop. The usual failure is termination — the model decides it's "done" in prose, then drifts.

Lagrange makes scoring and stopping first-class. The loop is an explicit state machine with pluggable slots. The kernel owns convergence, constraints, budget, and observability.

The name is literal: constraints carry a price

A loop that maximizes success under budgets (tokens, cost, wall-clock) and hard limits (tests pass) is constrained optimization. Lagrange optimizes the Lagrangian directly:

L = success − Σi λi · max(0, costi − limiti)

Each constraint owns a multiplier λ — the price of violating it. A PID controller updates λ from measured violation (integral term = dual ascent). The penalized objective drives selection and plateau detection. Over-budget candidates lose rank; they aren't hard-banned.

Honest about the metaphor

Constraint-pricing is literal and load-bearing. The full controller-theory machinery — stability guarantees, auto-converging λ — is not. λ is sensitive and task-dependent. Lagrange ships the controller and Pareto-sweep tooling, not a self-tuning knob.

Worked example: pass the tests within a diff budget

A Claude coding agent fixes a failing test suite under a diff budget. The goal is not just "tests pass" — it's "tests pass while changing at most 8 lines." That split is the point. The agent passed all 6 tests on turn 1, but with a 12-line diff. The constraint kept the loop going until the change fit the budget: 12 → 10 → 8 lines over three turns. Real run, throwaway repo, $0.74 total.

Why explicit constraints, not a prompt

"Keep the change small" in the system prompt is advice the model can rationalize away. A constraint is a measured number: the diff is counted every turn, written into the next prompt, and logged for audit. Here it is also a hard gate — convergence requires tests pass and diff ≤ budget — so the agent can't declare victory at turn 1 with a sprawling edit, and can't game the gate by editing the tests.

Show the full configuration 
from lagrange import Budget, Constraint, Score
from lagrange.constraint import LambdaController
from lagrange.core import Verdict
from lagrange.backends.claude import ClaudeExecutor
from lagrange.stop import Success, Plateau
from lagrange.verify import Verifier

REPO, CAP = "/path/to/project", 8.0

def objective(turn, ctx):              # score: value + the metric to constrain
    passed, total = run_pytest(REPO)   # parse pytest -> (passed, total)
    diff = changed_lines(REPO)         # git diff --numstat -> int
    return Score(value=passed / total, metrics={"diff": float(diff)})

class TestsAndBudget(Verifier):        # convergence = tests pass AND diff <= cap
    def verify(self, candidate, ctx):
        passed, total = run_pytest(REPO)
        diff = changed_lines(REPO)
        ok = passed == total and diff <= CAP
        return Verdict(passed=ok, detail=f"pytest {passed}/{total}, diff {diff}/{int(CAP)}")

def refine(s):                         # surface the constraint to the agent
    m = s.ctx.last_candidate.score
    diff = int(m.metrics["diff"])
    if m.value < 1.0:
        return f"Tests still failing. Fix stats.py; diff budget is {int(CAP)} (at {diff})."
    return (f"All tests pass, but your diff is {diff} lines; the budget is {int(CAP)}. "
            f"Make the edits minimal — no rewrites or extra helpers. Do not edit the tests.")

executor = ClaudeExecutor(
    task="Fix the bugs in stats.py so the test suite passes.",
    goal="pytest passes within an 8-line diff budget", cwd=REPO,
    allowed_tools=["Read", "Edit", "Bash", "Glob", "Grep"],
    refine=refine, per_step_max_turns=12,
)

loop = executor.build_loop(
    objective=objective,
    verifier=TestsAndBudget(),
    constraints=[Constraint("diff", limit=CAP,
                            controller=LambdaController(ki=0.05))],
    stop=Success() | Plateau(patience=4),
    budget=Budget(max_iters=6, max_cost_usd=6.0),
)
result = loop.run()                    # converged in 3 turns, $0.74

What happened: turn 1 fixed every test but changed 12 lines, over the budget, so the gate held and λ rose. The next prompt told the agent its diff and the budget. It tightened to 10 lines, then to 8 — the budget — and the loop converged. The naive goal ("tests pass") was met at turn 1; the constrained goal took three.

The stop algebra refuses underspecified loops

Stopping is a composable algebra. Conditions combine with |; the first to fire wins and records why — and whether it converged.

stop = Success() | Plateau(patience=3) | Oscillation() | MaxIters(20)

Each condition declares two facts: can it converge (success), does it guarantee termination (a hard cap). The loop refuses a config that can't do both. A loop that only stops by exhausting its budget is the lineage's costliest anti-pattern.

One query() call is one loop step

The default backend drives a Claude agent. One step = one query() turn. The executor chains session_id, so context carries. A failed verdict becomes the next prompt — a refine cycle. Per-turn caps map to max_turns / max_budget_usd. Per-turn usage feeds the loop, so TokenBudget / CostBudget bound the whole run.

Show the coding-loop in full 
from lagrange import Budget
from lagrange.backends.claude import ClaudeExecutor
from lagrange.stop import Success, Plateau
from lagrange.verify import CommandVerifier

executor = ClaudeExecutor(
    task="The tests are failing. Fix the code and run `pytest -q`.",
    goal="pytest passes",
    cwd="/path/to/project",
    allowed_tools=["Read", "Edit", "Bash"],
    per_step_max_turns=30,          # SDK hard stop per turn
)

loop = executor.build_loop(
    verifier=CommandVerifier(lambda _: ["pytest", "-q"], cwd="/path/to/project"),
    stop=Success() | Plateau(patience=3),
    budget=Budget(max_iters=8, max_cost_usd=10.0),
)
result = loop.run()             # result.best.payload.text == final agent output

Two verifiers. CommandVerifier runs a command and gates on the exit code — deterministic, transcript-provable. ClaudeJudge is the actor/critic split: a fresh cheap model judges the agent's surfaced output. It can't run tools, so the goal must be provable from what the agent showed.

One checkpoint primitive: resume, time-travel, HITL

Give a loop a run_id and a Persister. It checkpoints every iteration. One primitive, three uses:

  • Crash recovery — restart, call run() again. The loop restores iteration, archive, memory, λ state, usage, and the agent's session_id, then continues.
  • Time-travel — every iteration is kept. history(run_id) lists them; load(run_id, n) loads any.
  • Human-in-the-loop — a Pause condition checkpoints and exits. Release it and resume.

Two stores: MemoryPersister (in-process) and SQLitePersister (durable, stdlib-only, survives a crash). The Claude executor auto-registers as a state provider — the agent session is checkpointed with no extra wiring.

Limitations

  • SQLitePersister uses pickle — never load checkpoints from an untrusted source.
  • The ClaudeJudge verifier can't run tools; convergence conditions it judges must be provable from the agent's surfaced output.
  • Each step bridges sync↔async with asyncio.run; it raises if called from inside a running event loop (async API is future work).
  • λ is sensitive and task-dependent — the controller helps, but tuning is still empirical.
